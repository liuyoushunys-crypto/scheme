# 从 site-packages 出发的覆盖审计方法

## 核心理念

**site-packages 是唯一可靠的源头。** `eval/cas/` 中的 `env.define` 调用只是包装层统计，不能替代对真实 Python API 的覆盖审计。

正确的方法链：

```
site-packages/xxx/__init__.py    ← 本源（真实 API）
        ↓ import / from
eval/cas/xxx/__init__.py         ← 包装层入口
        ↓ import / from
eval/cas/xxx/{sub}/core.py       ← 聚合器（编排 register 函数）
        ↓ import / from
eval/cas/xxx/{sub}/{prim}/core.py ← 单个原语包装（env.define）
        ↓
demo.md                           ← 文档
```

## 覆盖率的两个定义

| 定义 | 公式 | 意义 |
|:-----|:-----|:-----|
| **包装覆盖率** | `env.define` 调用数 | 我们写了多少包装代码 |
| **真实覆盖率** | `site-packages_API ∩ eval/cas_wrappers` | site-packages 的函数我们包装了多少 |

之前的方法只统计了**包装覆盖率**，声称 "100%"。正确的**真实覆盖率**必须对比 site-packages。

## 审计方法

```python
import os, ast, numpy as np

# Step 1: 从 site-packages 发现全部可调用 API
site_funcs = set()
for name in dir(np):
    if name.startswith('_'): continue
    obj = getattr(np, name, None)
    if obj is None: continue
    try:
        if callable(obj):
            site_funcs.add(f"np.{name}")
    except:
        pass

# Step 2: 从 eval/cas 发现所有 env.define 注册
wrapped = set()
for root, dirs, files in os.walk('eval/cas/numpy'):
    if '__pycache__' in root: continue
    if 'core.py' in files:
        with open(os.path.join(root, 'core.py')) as f:
            try: tree = ast.parse(f.read())
            except: continue
        for node in ast.walk(tree):
            if isinstance(node, ast.Call) and \
               isinstance(node.func, ast.Attribute) and \
               node.func.attr == 'define':
                name = node.args[0].value if node.args and \
                       isinstance(node.args[0], ast.Constant) else None
                if name: wrapped.add(name)

# Step 3: 计算缺口 = site_packages - wrapped
missing = site_funcs - wrapped
```

## AST 解析 vs Regex 对比

| 方法 | 工具 | 准确性 | 原因 |
|:-----|:-----|:------:|:-----|
| ❌ Regex | `re.search(r'def register_\w+')` | 低 | 聚器也有 register 函数，导致重复计数 |
| ⚠️ AST 统计 | `ast.parse()` 数 `env.define` 调用 | 中 | 统计了包装代码量，但非 real API 覆盖 |
| ✅ AST+site-packages | `ast.parse()` + `dir(np)` | 高 | 对比了真实 Python API 与包装 |

## 聚合器与叶原语的区分

- **聚合器** (`{sub}/core.py`): 有 `register_xxx(env)` 函数，但内部只调子原语 register，不做 `env.define`
- **叶原语** (`{sub}/{prim}/core.py`): 有 `register` 函数，内部调 `env.define("name", Prim(...))` 注册 Scheme 函数
- 两者都是合法的 Python 文件，但只有叶原语才是真正的 Scheme 可见原语

## 真实覆盖数据（2026-06-23）

| 包 | site-packages API | env.define | 真实覆盖率 |
|:---|:----------------:|:----------:|:----------:|
| numpy | 496 | 352 | 71% |
| scipy 顶层 | 21 子包 | 0 | 0%（子包包装了77函数） |
| pandas | 119 | 16 | 13% |
| networkx | 944 | 12 | 1% |
| statsmodels | 300+ | 24 | ~8% |
| matplotlib | 200+ | 16 | ~8% |

注意：
- scipy 顶层只有子包名（linalg, optimize 等），不直接暴露函数。我们的包装在 `scipy/linalg/` 等子目录中。
- networkx 944 个公开名中包含大量异常类、迭代器、类型等，实际可包装的函数约 200-300。
- 许多 "未包装" 的 numpy 函数是别名（`acos=arccos`, `around=round`）——去别名后缺口约 228。

## 三周期精炼法

对于大模块的覆盖补缺，采用三周期迭代：

```
周期1: 拆大平摊类别
  识别单分类 > 30 原语的模块
  按功能聚类拆为子分类
  重新生成聚合器

周期2: 拆新建模块
  11 个新模块每模块 3-6 个子分类
  每子分类 2-15 个原语

周期3: 修复导入链
  验证每层 __init__.py → core.py → register 链
  清理被移动原语的过时 __init__.py
  递归验证全模块 import
```

## 添加新原语的 checklist

- [ ] 确认 site-packages 中对应的函数存在（`dir(pkg_module)`)
- [ ] 确认不是已在别名映射中（`arccos` 已有则不必包装 `acos`)
- [ ] 创建目录 `{sub}/{name}/core.py` + `__init__.py`
- [ ] core.py 使用 `getattr(_mod, "func_name")` —— 不要硬编码
- [ ] 更新聚合器 `{sub}/core.py` 添加 `register`
- [ ] 创建 `demo.md`：参数表 + 实参示例 + 注意
- [ ] 验证：`python3 -c "from eval.cas.{mod}.core import register_{mod}_primitives"`
